# 全链路 QC — 一致性是过程的产物

> 传统做 IC Memo 或写 research report 时，质量控制是最后一步——写完了回头检查数字对不对、来源标了没有、前后有没有矛盾。这个过程**痛苦且容易遗漏**。
>
> 这套系统的 QC 不是事后的，**是嵌在过程里的**。

## 五层 QC

### 1. 数据入口 QC

每个数据点进入 wiki 时就带着**来源血缘**：

- 从哪个文件的哪一行提取的
- 是 A 级（财报 / 模型原始数据）还是 B 级（实地调研纪要 / 管理层口径 / 卖方推算 / web search）

**分级在入库时就确定，不是用的时候才想起来标。**

### 2. 注册表 QC

模型数字通过 **registry** 追踪：

- 每个数据点注册在 registry 里，记录 Excel 模型的 tab 和行号
- 模型更新后自动 diff，变化列出来让你确认
- wiki 上不会出现"模型更新了但页面还是旧数字"的情况

### 3. 格式 QC（12 条硬规则）

12 条硬规则覆盖所有踩过的坑：

1. 表格不能有空单元格
2. 来源引用必须是 wikilink（不允许裸文本来源）
3. 年度和季度数据不能混在同一张表
4. 估值指标必须带"截止日期 + 基准股价"标注
5. 关键数字必须有来源或口径备查，不在正文堆标签化来源分级
6. 管理层口径必须识别 talk-book 风险，不能直接当事实
7. 跨段落引用同一数字必须一致（不允许同页两个不同数）
8. §6 Known Truth 改动必须在 §13 Diff Log 留 diff
9. Q&A 存档必须带置信度
10. 催化剂日历必须带"如果 deliver / miss 各意味着什么"
11. 资料缺口只做内部记录；仅在影响核心结论时写入正文
12. 合规声明必须出现在末尾

每次更新自动 **lint**，不通过不算完成。

### 4. 时效 QC

健康检查每天巡查六项条件，TP / 研报 / 催化剂 / 财报日期全覆盖。详见 [health-check.md](health-check.md)。

### 5. 交叉 QC

vault 扫描生成 question list 时，主动对比不同来源的数字，**差异超 5% 就标记**。

## 关键观察

**这些 QC 分布在研究流程的每个阶段**，不需要单独花时间做"质量检查"。到出 IC Memo 或做 PPT 的时候：

- 数据已经是**验证过的**
- 来源已经是**标注过的**
- 前后一致性已经是**保证过的**

所有研究过程**清晰可见**：

- wiki page 上每个数字可追溯来源
- 每次更新有 diff 日志
- 每轮 Q&A 有完整记录
- 每天的信息流有时间戳

这意味着最终生成 PPT 或 Word 时，系统直接从 wiki page 拉数据、引用 wikilink 作为来源注释——**而这些数据已经经过入库、注册、交叉验证、时效检查的全链路 QC**。

## 最终输出质量是过程的自然结果

**最终输出物的质量和一致性，是过程的自然结果，不是事后修补的产物。**

这是为什么传统 workflow 里"做 deck"是一个独立的、痛苦的、容易出错的阶段。**在新的系统里，做 deck 只是从 wiki page 抽取已验证的数据和结论，选择呈现方式**——信息收集、交叉验证、来源标注这些脏活在过去的日常跟踪中已经做完了。这就是为什么最终输出的 PPT 和 Word 的质量和一致性会非常高——**不是因为最后一刻仔细，是因为整个过程里就没有妥协。**

## 与原有 acceptance.md 的关系

- `acceptance.md`：**输出层**验收清单（写完了之后逐条自检）
- `full-qc.md`：**过程层** QC 嵌入（写的时候就在跑）

两者互补，不冲突。Librarian 模式下 full-qc 是默认；非 Librarian 模式（手工写报告）下 acceptance 是兜底。
