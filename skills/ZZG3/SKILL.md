# ZZG3 事件驱动投资机会分析 Skill

## 最高优先级

用户给出一段“变化”并问有哪些公司值得看、受益、受损、更有投资价值，或者问“这件事对某家公司有什么影响”“这个业绩怎么看”时，本 skill 是默认入口。不要把它降级成新闻摘要，也不要只给一串股票名。ZZG3 的任务是先判断这是不是事件驱动问题：只要存在明确事件或边际变化，并且用户关心影响、利好利空、公司筛选或业绩解读，就按事件驱动处理。

“值得投资”必须改写为“在什么前提、周期和风险收益画像下更占优”。不得输出无条件买入/卖出指令，不得写“必涨”“确定性机会”。

## 与旧 skill 的覆盖关系

ZZG3 覆盖以下旧入口的交叉场景：

- `sm-catalyst-monitor`：当用户不只是要跟踪催化，而是要从事件推导投资标的时，改走 ZZG3。
- `sm-stock-screen`：当筛选起点是具体事件、公告、政策或行业边际变化，而不是泛主题挖掘时，改走 ZZG3。

保留旧 skill 的边界：

- 只问“这个催化剂对 X 有什么影响”“未来一周怎么跟踪”，且不要求找投资候选公司时，可用 `sm-catalyst-monitor`。
- 只问“AI 链还有哪些方向”“涨得少的板块有哪些”，且没有具体事件材料时，可用 `sm-stock-screen`。
- A 级候选需要单家公司深度研究时，后续必须切到 `ZZG`。
- 多个 A 级候选需要横向比较时，后续必须切到 `ZZG2`。
- 事件本质是行业长期框架变化，且用户要完整行业研究时，后续必须切到 `ZZG1`。

## 信息源纪律

在任何定性判断前，必须先完成资料读取和结构化校验。用户贴的材料只能作为事件线索，不能作为唯一证据。

### 0. 事件驱动判定

先判断用户问题是不是事件驱动。以下情况默认视为事件驱动并使用 ZZG3：

- “这件事对 X 有啥影响 / 对 X 是利好还是利空”
- “这件事对哪些公司利好 / 哪些公司受益 / 哪些公司受损”
- “这个业绩怎么看 / 业绩出来怎么看 / 业绩超预期或低于预期怎么看”
- “这个公告、订单、价格、政策、监管、产品发布、技术突破怎么看”
- “这段信息怎么选股 / 哪些票值得看 / 哪些公司值得优先研究”

不走 ZZG3 的边界：

- 用户明确要财报前瞻、业绩前瞻、earnings preview，且财报尚未披露：走 `sm-earnings-preview`。
- 用户明确要完整单家公司深度，不是事件影响：走 `ZZG`。
- 用户明确要完整行业报告：走 `ZZG1`。

### 1. 事件识别

先把用户材料拆成结构化事件：

- 事件类型：政策 / 监管 / 订单 / 价格 / 成本 / 产能 / 库存 / 产品发布 / 技术突破 / 产业数据 / 业绩/财报结果 / 管理层表态 / 地缘政治 / 其他。
- 变化变量：需求、供给、价格、成本、份额、资本开支、库存周期、渠道、估值锚、风险偏好。
- 方向与强度：利好 / 利空 / 中性；一次性扰动 / 趋势性变化；影响窗口是日内、一周、一季度、一年还是多年。
- 产业链位置：上游、中游、下游、设备、材料、零部件、渠道、客户、替代品、竞争对手。
- 待核验点：哪些信息来自用户材料，哪些必须用公告、财报、研报或结构化数据确认。

### 2. 近 5 年卖方研报先行

识别相关行业和初步候选公司后，在任何结论前读取近 5 年卖方 / 券商研究报告正文或 PDF：

- 行业研报：行业深度、产业链专题、供需格局、价格周期、政策专题、技术路线、竞争格局。
- 公司研报：公司深度、首次覆盖、年报点评、季报点评、跟踪报告、事件点评。
- 优先东方财富 / dfcfw 报告正文、PDF 原文、转载正文。
- 不得只看摘要、评级、目标价、PE 表或盈利预测摘要。

如果找不到足够正文 / PDF，或只能拿到摘要，必须在 `[Preflight]` 的缺口记录中列明；最终报告只在该缺口影响核心结论时简短说明。

### 3. 结构化数据校验

按 Investor Harness 当前规则调用同花顺 iFind MCP / 已配置金融数据能力，校验：

- 公告、定期报告、临时公告、业绩预告、订单和合同披露。
- 行情、市值、估值、成交、涨跌幅、相对行业表现。
- 财务指标：收入、利润、毛利率、费用率、现金流、资产负债、资本开支。
- 股东、机构持仓、回购、减持、增发、限售解禁。
- 事件、新闻、行业指标、价格数据、产能数据、库存数据。
- 港股和美股使用 global_stock / 全球股票结构化数据工具。

### 4. 补充搜索

完成研报和结构化数据后，再使用：

- `cn-web-search`
- broader WebSearch
- 公司官网 / IR / 交易所 / SEC / 港交所等官方披露
- 用户贴的材料

补充搜索只用于确认最新事实、口径和时间线，不能替代研报正文、公司披露和结构化数据。

## 分析框架

### 1. 事件到盈利变量

必须写清楚传导链：

```text
事件 / 变化
-> 需求、供给、价格、成本、份额、资本开支、库存或风险偏好变量
-> 公司收入、毛利率、费用率、利润、现金流或估值锚
-> 一致预期是否需要上修 / 下修
-> 股价验证窗口与关键数据点
```

### 2. 候选公司分层

候选池必须至少分四类：

- A 级优先研究：事件相关度高、盈利弹性可验证、预期差可能存在、风险收益画像较好。
- B 级观察：逻辑成立但缺关键数据，或已经部分定价。
- C 级备选：二阶受益、弹性较弱、信息不足，暂不深挖。
- 排除 / 受损清单：弱相关、纯概念、反向受损、数据无法验证或估值已经过度反映。

### 3. 评分口径

对每个候选公司给 0-100 分，并解释核心扣分项：

| 维度 | 权重 | 判断要点 |
|---|---:|---|
| 事件相关度 | 25 | 业务收入、客户、成本、供应链或产品是否直接暴露 |
| 盈利弹性 | 20 | 对收入、毛利率、利润或现金流的边际影响 |
| 证据强度 | 15 | 是否有公告、财报、结构化数据、研报正文或多源交叉验证 |
| 预期差 | 15 | 市场是否尚未充分定价，未来 1-2 个季度是否有验证点 |
| 估值与交易拥挤度 | 10 | 估值是否透支、成交和持仓是否拥挤 |
| 风险调整 | 15 | 政策、价格回落、客户集中、技术替代、流动性和反身性风险 |

### 4. 后台可靠性判断

每条关键事实和公司入池理由都必须先做后台可靠性判断，但最终正文不输出标签化来源分级，也不设置“证据等级”列。常规高可信数据直接进入结论；只有弱来源、口径冲突、无法交叉验证的信息，或模型自行估算/测算的数据，才在相邻文字里短句说明来源、口径或“估算/测算”。

## 输出结构

最终报告标题统一为：

```markdown
# ZZG3 事件驱动投资机会分析：{事件或变化主题}
```

正文必须使用以下结构：

### 1. 结论先行

用一张表给出 A/B/C 分层和排除清单。不要超过 8 个核心候选，避免堆名单。

| 分层 | 公司 | 代码 | 市场 | 方向 | 分数 | 核心理由 | 最大疑问 | 下一步 |
|---|---|---|---|---|---:|---|---|---|

### 2. 事件拆解

说明事件是什么、变化变量是什么、影响窗口是什么、需要核验什么。

### 3. 影响路径

画出从事件到收入、利润、估值或情绪的传导链。必须区分一次性影响和趋势性变化。

### 4. 产业链受益 / 受损地图

分上游、中游、下游、替代品、竞争对手和客户，说明谁受益、谁受损、谁只是情绪相关。

### 5. 候选公司分层分析

逐家公司说明：

- 为什么入池。
- 受益或受损变量是什么。
- 对盈利或估值的可能影响。
- 当前是否已经被市场定价。
- 未来 1 周、1 个月、1 个季度的验证指标。
- 最大待核验点；如涉及弱来源、口径冲突或自行估算，在相邻文字里说明。

### 6. 排除清单

列出看似相关但不应优先研究的公司，说明排除原因：业务相关度弱、收入占比低、纯概念、数据不可验证、估值透支、事件反向受损等。

### 7. 验证节奏

列出未来需要跟踪的公告、价格、订单、出货、库存、政策细则、行业数据、财报科目、卖方一致预期变化和股价相对表现。

### 8. 反方与风险

必须至少列 3 条反方逻辑，并写清楚什么数据会证伪本次事件驱动机会。

### 9. 后续工作流

给出下一步 skill：

- A 级候选单家公司：`ZZG`
- 多家公司横向比较：`ZZG2`
- 行业长期变化：`ZZG1`
- 预期差验证：`sm-consensus-watch`
- 反方深挖：`sm-red-team`
- 只做后续催化跟踪：`sm-catalyst-monitor`
